Since 1976

We already measure this… right?

McCabe's Cyclomatic Complexity: the number of independent paths through the code — every function starts at 1, every branch adds 1. 


def parse_config(path):
    config = {}
    if not path.exists():
        return config
    for line in path.read_text().splitlines():
        if "=" in line:
            key, value = line.split("=", 1)
            config[key] = value
    return config
CC = 4 → you need 4 test cases for full branch coverage. Perfect for testability. But is it what makes code hard to read?
First use of the counter — explain the visual: badges land on lines, the box on the right keeps the running total. CC here is the baseline everyone knows; we build the critique on top of it.

2017 · SonarSource

Cognitive Complexity

Proposed by G. Ann Campbell: a score for how hard code is to understand — built from programmer intuition, not from a mathematical model.

"Ignore structures that allow multiple statements to be readably shorthanded into one. Increment for each break in the linear flow. Increment when flow-breaking structures are nested." — the three rules, Cognitive Complexity white paper

That's the whole spec. Let's watch it run on real code.

Don't explain the rules abstractly — announce that we'll discover them by scoring the two functions we just saw.

Rule discovery #1

Scoring sum_of_primes


def sum_of_primes(max):
    total = 0
    for i in range(1, max + 1):
        is_prime = True
        for j in range(2, i):
            if i % j == 0:
                is_prime = False
                break
        if is_prime:
            total += i
    return total
+1 for each break in the linear flow, +1 more per nesting level — the deeper you are, the more context you're holding in your head.
The signature moment — go slow. Call out the nesting bars: the inner if costs 3, not 1, because you read it while holding two loops in your head. And the break costs nothing: early exits make code clearer.

Why nesting?

Five structures, two shapes

Linear — one thing at a time


def process(items):
    for i in items:       # +1
        check(i)
    for j in items:       # +1
        if j.valid:       # +2
            use(j)
    if items:             # +1
        init()

Nested — everything at once


def process(items):
    for i in items:            # +1
        for j in items:        # +2
            if j.valid:        # +3
                if j.active:   # +4
                    use(j)
    if items:                  # +1
        init()

Campbell: "a linear series of five structures is easier to understand than those same five structures successively nested — regardless of the number of execution paths." Both have the same CC.

Rule discovery #3

Booleans: mixing is what costs

One sequence of like operators


def ready(a, b, c, d):
    # and, and, and → one sequence
    return a and b and c and d
1
cognitive

Three sequences — and / or / and


def ready(a, b, c, d):
    # every and-or switch starts a new one
    return a and b or c and d
3
cognitive
+1 per sequence, not per operator — a and b and c and d reads fine; it's the and ↔ or switches that make you stop and think.
CC charges per operator; CogC discounts uniform chains. Yet another case where the metric encodes taste you already have.

The philosophy

What costs nothing

+0

Early exits

return, break, continue — leaving early usually makes code clearer.

+0

Safety nets

with, try / finally — only except adds a branch (+1 each).

+0

Shorthand

Extracting well-named functions, comprehensions as expression shorthand — readable condensation is free.

"It should incent good coding practices: ignore or discount features that make code more readable." — the design principle behind every +0
This is the part cyclomatic gets backwards — CC often *punishes* the refactors that make code clearer. CogC is designed so that the score drops exactly when readability improves.

A PyCon exclusive, sort of

The paper has a Python appendix

Appendix A grants exactly three language exceptions. One of them is ours: decorators. 


def log_calls(func):
    def wrapper(*args):       # nesting stays at 0 — decorator exception
        if args:              # +1, not +2
            print("called with args")
        return func(*args)
    return wrapper            # total: 1
A function holding only a nested function and a return is a structural pattern, not logic — so the wrapper doesn't count as nesting. Add one more statement to log_calls and the exception is gone.
Nice trivia beat: the spec is language-agnostic but Python decorators were common enough to earn a carve-out. The exception is deliberately narrow — one extra statement in the outer function and the wrapper counts as level 1 again.

Refactor 1 / 4

Nested conditionals

Cognitive11

def get_discount(user, cart):
    if user is not None:            # +1

Refactor 1 / 4

→ collapse the tail

Cognitive4−64%

def get_discount(user, cart):
    if user is None:                        # +1
        return 0.0
    if not user.is_active:                  # +1
        return 0.0
    if cart.total <= 100:                   # +1
        return 0.0

Same behavior, and the rule reads top to bottom: no user? nothing. inactive? nothing. small cart? nothing. otherwise premium decides.
11 → 4 in four small, safe edits. The early returns are free; the flattening is what pays.

Refactor 2 / 4

→ flatten the filters

Cognitive4−60%

def active_premium_emails(users):
    result = []
    for u in users:                                    # +1
        if u.is_active and u.is_premium and u.email:   # +2, +1
            result.append(u.email.lower())
    return result
Three nested ifs become one and-chain: +2 for the if, +1 for the whole uniform and-sequence. That single edit is already 10 → 4.

Refactor 2 / 4

→ a comprehension

Cognitive3−70%

def active_premium_emails(users):
    return [
        u.email.lower()
        for u in users                                  # +1
        if u.is_active and u.is_premium and u.email     # +1, +1
    ]
The comprehension drops the loop's nesting level, so the same filter now costs +1 instead of +2.
"Shorthand multiple statements readably into one" — exactly what rule 1 was written to reward.

Flag + break

Cognitive6

    for item in items:                    # +1

Refactor 3 / 4

→ flatten the filters

Cognitive4−33%

def has_expired_item(items):
    found = False
    for item in items:                                      # +1
        if item.status == "active" and item.days_left < 0:  # +2, +1
            found = True
            break
    return found
Collapse the two conditions into one and-chain first: one if, one and-sequence. 6 → 4 before we even reach for a builtin.

Refactor 3 / 4

→ any()

Cognitive2−67%

def has_expired_item(items):
    return any(
        item.days_left < 0
        for item in items                 # +1
        if item.status == "active"        # +1
    )
any() and all() say "does one exist" / "do they all" directly — no flag, no break, no nesting.
6 → 4 → 2. Land the pattern on the next slide.

Refactor 4 / 4

Nested dispatch

Cognitive7

def run(command):
    parts = command.split()
    if parts[0] == "move":          # +1
        if len(parts) == 3:         # +2
            return f"move to {parts[1]},{parts[2]}"
    elif parts[0] == "rotate":      # +1
        if len(parts) == 2:         # +2
            return f"rotate {parts[1]}"
    elif parts[0] == "quit":        # +1
        return "bye"
    return "?"
Branching on the shape of the input: which command, how many arguments. Every "how many" check adds another nested if.

Refactor 4 / 4

→ match the pattern

Cognitive1−86%

def run(command):
    match command.split():          # +1
        case ["move", x, y]:
            return f"move to {x},{y}"
        case ["rotate", angle]:
            return f"rotate {angle}"
        case ["quit"]:
            return "bye"
        case _:
            return "?"
One case per shape checks length, position and value at once — the indexing and the nested guards all vanish.
7 → 1. The other three peeled nesting away by hand; here the right construct erases it in one move. When you're branching on the shape of data, match is built for exactly that.

None of these got shorter by accident.
They got flatter.

Remove nesting, not lines — that's what moves the number.

11→4, 10→3, 6→2, 7→1. In every case the win came from removing depth, and the metric agreed because it was designed to.

Takeaways

Three things to bring home

01

Cyclomatic counts paths, cognitive counts effort. Equal CC can hide wildly different reading costs — measure what you actually care about.

02

Nesting is the dominant cost. Guard clauses, comprehensions, any()/all(), extracted functions — flatten first.

03

Automate the measurement. A threshold in CI turns "this feels unreadable" into a number nobody has to argue about.

Simple code isn't short code — it's code you understand on first read.
